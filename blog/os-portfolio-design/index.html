The Idea

Most portfolios are a long scroll - hero, skills grid, project cards, contact. Fine, but forgettable. I wanted mine to make someone stop and explore. The concept: what if your portfolio was an operating system?

The visitor boots into a dark-themed desktop, then launches windowed apps to explore your work - just like opening apps on a real machine. It's interactive, memorable, and gives every section its own dedicated space without competing for attention on one long page.

Live: Explore the full UI at sreedevss.in. Source code is on GitHub.

Tech Stack

  • React 19 - functional components, hooks everywhere
  • TypeScript in strict mode throughout
  • Vite 7 - fast dev server and optimized production builds
  • Framer Motion 12 - all animations, springs, and AnimatePresence
  • Tailwind CSS 3 - utility-first styling with a custom design system
  • Context API - four contexts: Window, Music, Theme, and NebulaOverride
  • Web Audio API - real-time frequency analysis for the visualizer
  • Canvas API - the Nebula Override game and visualizer bars
  • EmailJS - contact form email delivery without a backend
  • gh-pages - deployment directly to GitHub Pages

The Boot Screen

The very first thing the visitor sees is a terminal-style boot log - timestamped system messages rolling out in a monospace font on a near-black background. It sets the tone immediately.

Fake authenticity through variable timing

The log prints 43 messages, but they don't appear at a fixed interval. Each line is delayed by Math.random() * 50 + 10 ms, giving it the feel of actual I/O. Specific lines (idx 3, 10, and 18) get an extra 500 ms pause - exactly where a real build tool would stall on something expensive, like transforming modules or mounting a filesystem. The log goes through realistic stages: cloning, npm build, Vite transforms, and finally "System READY." 

// Variable speed - fast bursts with deliberate pauses
let delay = Math.random() * 50 + 10;
if (idx === 3 || idx === 10 || idx === 18) delay += 500;

Skip mechanism with a safety gate

You can skip by clicking anywhere or pressing any key - but only after 30 lines have printed. A skipReady ref (not state, to avoid re-renders) is checked by both the keyboard handler and the click handler. This prevents accidental skips if someone clicks within the first second of loading.

Pre-fetching data during the boot

The boot screen isn't idle time - it uses a useEffect to kick off a projectService.fetchProjects() call to the GitHub API in the background. By the time the user opens the Projects app, the data is already cached. 

// App.tsx - boot → desktop transition
const App: React.FC = () => {
  const [bootComplete, setBootComplete] = useState(false);

  // Reveal SEO content once boot is done
  React.useEffect(() => {
    if (bootComplete) {
      const seo = document.getElementById('seo-content');
      if (seo) seo.style.display = 'block';
    }
  }, [bootComplete]);

  return (
    <AnimatePresence>
      {!bootComplete ? (
        <BootScreen key="boot" onComplete={() => setBootComplete(true)} />
      ) : (
        <Desktop key="desktop" />
      )}
    </AnimatePresence>
  );
};

The Desktop - Layers & Animation

The desktop is a fixed inset-0 div that hosts several visual layers, composited from back to front:

  1. Grid background - a CSS dot/grid pattern at low opacity
  2. Aurora blobs - two large radial-gradient blobs with blur, animated with a slow CSS keyframe. Crucially, they react to the music: when the Music App plays a track, the blob colors shift to match that track's color token within 1000–1500ms using a CSS transition.
  3. Vignette & scanlines - a radial gradient vignette and a repeating scanline overlay at 10% opacity for CRT realism
  4. TopBar - slides in from the top on mount with a spring animation
  5. Windows - rendered as a flat list from WindowContext
  6. Visualizer - a canvas overlay behind the Dock
  7. Dock - floats at the bottom-center
  8. NebulaOverride - mounted always, invisible when gameState === 'idle'

The staggered mount animation (TopBar slides in first at delay: 0.6s, then windows fade in at delay: 0.9s) gives the desktop a cinematic boot-to-ready feel, like a high-end OS coming online.

Commit 5f9ee44: "Updated animation" - this commit implemented the staggered desktop reveal, replacing jarring instant-appear with spring-physics-based sequence where the TopBar, window layer, and Dock each animate in at tuned offsets.

Window Management System

The WindowContext is the core of the OS illusion. It manages the full state of every open window: position, size, z-index, minimized state, and focus. 

// WindowContext.tsx - actual interface used in production
interface WindowState {
  id: string;
  title: string;
  icon: string;
  position: { x: number; y: number };
  size: { width: number; height: number };
  minimized: boolean;
  focused: boolean;
  zIndex: number;
}

Dragging & Resizing

Dragging and resizing are implemented via two custom hooks: useWindowDrag and useWindowResize. Both use the Pointer Events API for unified mouse/touch support. The title bar listens for onPointerDown to start a drag, and a resize handle in the bottom-right corner triggers resize mode. Double-clicking the title bar toggles maximize (with a pre-max state snapshot stored in React state for restore).

The Resume app is specifically locked - maximize is disabled for it (windowState.id === 'resume' guard) because it has a fixed layout designed for A4 printing and maximizing would break the print-preview.

localStorage Persistence

Every time a window is moved or resized, its position and size are written to localStorage under the key 'windowPositions'. On the next visit, the WindowProvider reads this on mount and merges the saved positions with the defaults. Your windows remember where you left them.

Minimization without unmounting

A key design decision: minimized windows are not unmounted. They're animated to opacity: 0, scale: 0.85, y: 30, display: 'none' with Framer Motion, keeping the React component fully alive in the tree. This means the Music Player keeps playing after being minimized, the Terminal keeps its session, and the Browser keeps its page - exactly like a real OS. 

// Window.tsx - minimize animates but doesn't unmount
animate={windowState.minimized
  ? { opacity: 0, scale: 0.85, y: 30, display: 'none' }
  : { opacity: 1, scale: 1, y: 0, display: 'block' }}
transition={{ type: 'spring', stiffness: 380, damping: 28 }}

Intelligent z-index management

A nextZIndex counter in the context starts at 100 and increments every time a window is focused or newly opened. focusWindow has a short-circuit: if the window is already at nextZIndex - 1 and already focused, it skips the state update entirely to prevent pointless re-renders.

Mobile responsiveness

On viewports ≤ 768px, any newly opened window is positioned at { x: 0, y: 32 } and sized to fill the full viewport height minus the TopBar and Dock - making it behave like a full-screen modal. No drag or resize handles are shown on mobile.

Commit b29b5b6: "Minimize works" - this was the commit that nailed the minimized state behavior, ensuring components stayed mounted and the dock dot correctly toggled between red (open) and amber (minimized).

The Dock - Interactions & State Indicators

The Dock is a glassmorphic bar centered at the bottom, containing 8 app icons, a brand tag, a theme toggle, and what appears to be a battery-saver button (more on that later).

Opening, toggling, and long-press to close

Each Dock button implements three behaviors via pointer event timing:

  • Tap/click → openWindow(). If the window is already focused, clicking again minimizes it. This creates a natural toggle.
  • Long press (600ms) → closeWindow(). A setTimeout is set on pointerDown and cancelled on pointerUp or pointerLeave. If it fires, the window is closed.

Active state indicators

When an app window is open, a dot appears below the Dock icon. The dot color tells the story: ■ red means the window is open and visible, ■ amber means it's minimized. The dot uses Framer Motion's layoutId so it animates smoothly when changing states.

The staggered entrance animation

The Dock uses variants with staggered children, so each icon pops up sequentially on first load - a spring-based entrance with y: 25 → 0, scale: 0.8 → 1, and staggerChildren: 0.04s.

The Apps - Each One In Detail

AboutApp - Typewriter and Scan-line Avatar

The About app opens with a hero section: a square avatar with animated red corner brackets, a scan-line that sweeps vertically through it using a CSS keyframe animation, and a typewriter that cycles through "Developer", "Designer", "Editor" at 90ms per character, with a 45ms backspace speed and a 2000ms pause between words. The typing cursor is a blinking 2px red bar.

Beneath that, skills are rendered as small tags with a left border colored by the technology's brand color (React gets #61dafb, Figma gets #f24e1e, etc.) - a detail that makes the grid feel alive at a glance.

ProjectsApp - Live GitHub API

Projects are fetched from the GitHub API using a projectService singleton. The service caches results so repeat opens don't re-fetch. Data is pre-fetched during the boot screen's idle time.

TerminalApp - Real Shell Simulation

The terminal is a genuine shell simulation, not just a pretty text box. It has:

  • Persistent virtual filesystem - stored in localStorage. You can mkdir folders, touch files, rm them, and cd into directories. The filesystem survives page refreshes.
  • Tab completion - pressing Tab completes commands or filenames. Multiple matches are printed as a list (just like bash).
  • Command history - ArrowUp/Down cycles through previous commands.
  • Async commands - matrix is an async command that streams lines with await delay(600), locking input while running.
  • Live GitHub integration - cd ~/projects && ls shows live repo names from the GitHub API. open <repoName> opens the repo in a new tab.
  • CRT aesthetic - a repeating-linear-gradient overlay creates scanlines on the terminal output, and green text has a subtle text-shadow glow.
  • Easter eggs: rick, cowsay, coffee, matrix.
Try it: In the terminal, type open browser, open about, or open music to launch any app directly from the command line.

ResumeApp - The Print Transformation

The CV is rendered in the OS's dark neon aesthetic. But printing neon-on-black as a PDF is unreadable. The handlePrint() function implements a full color transformation pipeline:

  1. Clone #resume-print-area into a temporary #print-mount div appended to <body>.
  2. Inject a <style> tag that remaps all neon shades to professional equivalents - for example, #00f0ff (cyan) becomes #0056b3 (deep blue).
  3. Inject a second style that enforces break-inside: avoid on every .relative.group experience block, preventing page cuts mid-entry.
  4. Call window.print().
  5. Remove both the injected mount and style tag in a setTimeout(100) cleanup.

ContactApp - EmailJS Without a Backend

The contact form uses @emailjs/browser to deliver messages directly to an email address from the browser, with zero backend required. The form validates all fields client-side before submission.

BrowserApp - In-OS Web Browser

A fully functional mini-browser window with an address bar, navigation controls, and an <iframe> viewport. It includes quick-launch bookmarks for common sites.

VSCodeApp - Easter Egg

The VS Code app is a lightweight easter egg - it embeds a link to an online code editor or redirects to the GitHub source. Part of the joke of running a full "OS."

Music Player & Web Audio Visualizer

The Player

The MusicApp is a two-column layout: a main player with an animated vinyl disc on the left, and a 30-track scrollable playlist on the right. The vinyl disc physically slides out (translateX(15%)) when playing, revealing the album art, and spins via a CSS animation: spin 8s linear infinite that pauses when music stops.

The player supports shuffle, repeat, volume control, and a seek bar. The "previous track" button has smart logic: if you're more than 3 seconds into a track, it restarts the track instead of going back - the same behavior as Spotify.

Per-track color theming across the whole OS

Every track has a color property (e.g., '#ff7e67' for Lofi Beats, '#9d00ff' for Deep Work). When a track plays, setMusicState pushes the color to the global MusicContext. The Desktop component reads this and uses it for the aurora blob radial gradients - so the entire desktop's ambient lighting shifts to match whatever you're listening to with a 1–2 second CSS transition.

The Web Audio API pipeline

The visualizer is powered by the Web Audio API. On the first user interaction (required by browsers to allow audio context creation), the initAudio() function builds a pipeline: 

// MusicApp.tsx - Web Audio API pipeline
const ctx = new AudioContext();
const analyser = ctx.createAnalyser();
analyser.fftSize = 1024;         // 512 frequency bins
analyser.smoothingTimeConstant = 0.85; // built-in temporal smoothing

const source = ctx.createMediaElementSource(audioRef.current!);
source.connect(analyser);
analyser.connect(ctx.destination);

// Share the analyser globally so the Visualizer component can read it
setMusicState(prev => ({ ...prev, analyserNode: analyser }));

The analyser reference is stored in MusicContext and shared across the app tree. The standalone Visualizer component reads it on every animation frame.

CORS bypass for audio analysis

The Web Audio API's createMediaElementSource requires the audio to be served with CORS headers. Public MP3 CDNs often block this. The solution was to proxy tracks through GitHub's raw content CDN (raw.githubusercontent.com) for a mini-player repository that explicitly allows CORS - bypassing 403 blocks from corsproxy.io entirely.

The Visualizer - AGC + Hybrid Indexing

Most web visualizers naively map frequency bins 1:1 to bars. This makes high-frequency bars almost invisible since most music energy is in the bass and mids. Two techniques fix this:

Hybrid Indexing - each bar's frequency bin is computed with a mixed linear+power curve: index = (i * 0.45) + Math.pow(fraction, 2.8) * (binCount * 0.6). This ensures bass bars are unique while mids and highs are still well-represented.

Auto-Gain Control (AGC) - instead of a fixed normalization ceiling, maxPeak tracks the loudest signal seen and slowly decays at 0.99× per frame (maxPeak * 0.99). Quiet tracks don't show tiny bars; loud tracks don't clip.

Lateral smoothing - each bar's target height is a weighted average of itself and its neighbors (60% self, 20% each side), removing the jagged sawtooth look.

Attack/release damping - bars snap up at 35% per frame (fast attack) but decay at only 8% per frame (slow release), mimicking the feel of a hardware VU meter.

Nebula Override - The Hidden Game

How to unlock it

The Nebula Override game is completely hidden. There's no hint in the UI. To activate it, click the battery-saver button in the Dock 5 times within 2 seconds. The NebulaOverrideContext tracks click timestamps in a ref and checks them against a 2000ms rolling window. Hit the threshold, and the game activates. 

// NebulaOverrideContext.tsx - secret trigger
const CLICK_THRESHOLD = 5;
const CLICK_WINDOW_MS = 2000;

const registerBatterySaverClick = () => {
  const now = Date.now();
  clickTimestamps.current.push(now);
  // Prune timestamps outside the window
  clickTimestamps.current = clickTimestamps.current.filter(
    ts => now - ts < CLICK_WINDOW_MS
  );
  if (clickTimestamps.current.length >= CLICK_THRESHOLD) {
    activateNebula();
  }
};

Activation sequence

Activating triggers a 2.5-second "activating" state that overlays glitchy terminal text on the desktop: "SYSTEM INTERRUPT DETECTED", "BYPASSING SECURITY PROTOCOLS...", "LOADING NEBULA_OVERRIDE.exe". The text is printed line by line with 350ms intervals, and a glitchIntensity value escalates to distort the screen. At 2000ms, the text fades out. At 2500ms, gameState transitions to 'playing'.

The game engine

The game is a full space-shooter written entirely with the Canvas API - no game engine, no library. It runs in a requestAnimationFrame loop and manages:

  • 12 enemy types - Scout, Fighter, Cruiser, Bomber, Interceptor, Carrier, Star, Tank, Dart, Ghost, Juggernaut, Seeker. Each has HP, speed, size, and a minimum wave when it starts appearing. The Ghost drifts sinusoidally; the Seeker homes in on the player.
  • Player ship - controlled via keyboard (WASD/arrows) or mouse. Mouse control uses smooth interpolation (player.x += (mouseX - player.x) * 0.3) for fluid movement.
  • 6 power-up types - Spread, Rapid, Wingman, Homing, Bomb, Shield, Life. Each drops from enemies with probability scaled by the enemy's HP value.
  • Weighted wave spawning - enemy type selection uses a weighted random algorithm: newer, tougher enemies get lower weight as they age, preventing later waves from being monopolized by one type.
  • Progressive difficulty - spawn interval shrinks from 1500ms to 250ms, and enemy count per wave scales as floor(10 + wave * 5 + wave^1.5).
  • Persistent high score - stored in localStorage('nebulaHighScore').
Secret: Click the battery icon in the Dock 5× rapidly. Keep looking - you'll find it.

Design System

The visual language is dark cyberpunk-minimal. Three accent colors over a near-black base:

  • #ff003c - Red accent. Primary brand color, used for active states, glows, and the MT logo.
  • #ffaa00 - Amber. Secondary actions, code highlights, warnings.
  • #00f0ff - Cyan. Links, section headings on the Resume, and the visualizer default color.

Three typefaces, each serving a role:

  • Space Grotesk 700 - Display headings. Bold, geometric, tight letter-spacing.
  • Inter 400–700 - Body text and UI copy. Legible at all sizes.
  • JetBrains Mono 400–500 - Labels, tags, timestamps, all monospaced elements. Provides the tech authenticity without overusing code formatting.

The window chrome mirrors macOS traffic lights - red close (#ff5f56), amber minimize (#ffbd2e), green maximize (#27c93f) - with glyph icons (✕, −, +) that only appear on group-hover, exactly like macOS.

The SEO Challenge

A pure-client SPA on GitHub Pages means Google's crawler initially sees only an empty <div id="root"></div>. Server-side rendering isn't an option on GitHub Pages.

The solution was layered. First, a legitimate visible SEO footer was added below the fixed-height OS viewport in index.html. It contains real text content (bio, skills, links) that appears when a user scrolls past the OS UI. Search engines can crawl it; real users rarely see it.

Second, a network of static HTML pages was built in the public/ directory: /about, /projects, and this /blog. These are served directly by GitHub Pages without any JavaScript - fully crawlable, fully linkable subpages that give Google real topical depth beyond the SPA root.

Commits 19efa8d + a5b3720: "Updated SEO" and "Updated SEO optimisations" - these two commits introduced the static subpages, sitemap.xml, and the schema.org structured data (BlogPosting, Person, WebSite) that now appears across all pages.

The index.html also contains full Open Graph and Twitter Card meta tags, a <link rel="canonical"> on every page, and JSON-LD structured data. The previously risky "hidden div" SEO approach was replaced entirely to avoid Google penalties.

Key Lessons

  • Context API is sufficient for orchestrated OS state. Four contexts cover the entire app: windows, music, theme, and the game. No Redux, no Zustand. The key is keeping each context focused and not making them aware of each other - communication is one-way through props.
  • AnimatePresence is non-negotiable for mount/unmount animations. Without it, components disappear instantly. With it, exit animations play in full before unmounting. Every window, every screen transition uses it.
  • Don't unmount minimized windows. Learned this the hard way - music stopped, terminal sessions reset. Keep them in the tree, hide them visually.
  • Web Audio API's CORS requirement is a trap. You cannot analyze audio from a URL that doesn't send CORS headers, no matter how you load it. Plan your audio hosting early.
  • Variable timing makes fake boot sequences believable. A fixed 80ms interval feels like a loading spinner. Random bursts with strategic pauses feel like a real system.
  • Static HTML pages in public/ are the pragmatic SEO answer for SPAs on GitHub Pages. They're served directly, require no JS, and give Google real content to index.
