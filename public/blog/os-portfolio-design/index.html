How I Built an OS-Style Portfolio with React & Framer Motion

A deep-dive into the architecture, animation strategy, and design decisions behind building an interactive operating system UI as a personal portfolio - boot sequences, windowed apps, context APIs, and more.

MT

Sreedev SS

April 2, 2026 · ~8 min read

React TypeScript Framer Motion Vite UI Design Portfolio

The Idea

Most developer portfolios follow the same playbook - hero section, skills grid, project cards, contact form. I wanted something that would actually make someone stop and explore. The concept: what if your portfolio was an operating system?

The visitor "boots" into a dark-themed OS desktop, then launches windowed apps to explore your work - just like opening apps on a real computer. It's interactive, memorable, and gives every section its own space without scrolling through a single long page.

Live demo: You can explore the full UI at sreedevss.in. The source code is on GitHub.

Tech Stack

  • React 18 with functional components and hooks
  • TypeScript throughout - strict mode enabled
  • Vite for the build tool and dev server
  • Framer Motion for all animations and transitions
  • Tailwind CSS for utility-first styling
  • Context API for window, theme, and music state

Architecture Overview

The app has three main layers that compose the full experience:

1. Boot Screen

When the user first visits, they see a cinematic boot sequence - a loading bar, blinking cursor, and system messages in a terminal font. This is handled by the BootScreen component which fires an onComplete callback once the animation finishes, triggering React's AnimatePresence to unmount the boot screen and mount the desktop. 

// App.tsx - boot → desktop transition
const App = () => {
  const [bootComplete, setBootComplete] = useState(false);

  return (
    <AnimatePresence>
      {!bootComplete ? (
        <BootScreen onComplete={() => setBootComplete(true)} />
      ) : (
        <Desktop />
      )}
    </AnimatePresence>
  );
};

2. Window Management System

The WindowContext is the core of the OS illusion. It tracks every open window - its position, size, z-index, minimized state, and which app it renders. Windows can be dragged, resized, minimized to the Dock, and brought to focus. 

// WindowContext shape
interface WindowState {
  id: string;
  appId: AppId;
  title: string;
  position: { x: number; y: number };
  size: { w: number; h: number };
  zIndex: number;
  minimized: boolean;
}

Each Window component uses Framer Motion's drag prop constrained to the desktop viewport. Z-index is managed by tracking the highest current value and incrementing on focus.

3. App Components

Each "app" is just a React component rendered inside the Window chrome. The apps include:

  • AboutApp - bio, animated typewriter role, skills, and social links
  • ProjectsApp - live GitHub repo cards pulled from the API
  • ResumeApp - interactive resume with print-to-PDF support
  • ContactApp - contact form with email integration
  • TerminalApp - functional terminal with custom commands
  • MusicApp - ambient music player with visualizer
  • BrowserApp - mini in-OS browser window

Animation Strategy

Every animation in the OS follows a consistent set of physics parameters. Windows enter with a spring-based scale-fade, the Dock icons have a macOS-style magnification hover, and the boot sequence uses staggered delays. 

// Shared spring config used across window enter/exit
const SPRING = {
  type: "spring",
  stiffness: 280,
  damping: 24,
  mass: 0.8,
};

// Window mount animation
initial={{ opacity: 0, scale: 0.92, y: 8 }}
animate={{ opacity: 1, scale: 1, y: 0 }}
exit={{ opacity: 0, scale: 0.88, y: -4 }}
transition={SPRING}

Design System

The visual language is dark cyberpunk-minimal. Everything is built on a near-black #0a0a0f base with a red accent (#ff003c), amber (#ffaa00), and cyan (#61dafb).

Typography uses three complementary fonts:

  • Space Grotesk for display headings - bold, geometric
  • Inter for body text - clean and readable
  • JetBrains Mono for labels, tags, and code - tech authenticity

The SEO Challenge

A JS-driven SPA meant Google initially saw an empty <div id="root">. Since switching to SSR wasn't viable for a GitHub Pages static site, I took a different approach: adding a real, visible footer section to the HTML that appears below the OS viewport when users scroll. It contains structured content (bio, skills, projects, contact) and links to static subpages.

These static pages (/about, /projects, /blog) live in the public/ directory, served directly by GitHub Pages without any JavaScript - giving Google real, crawlable topical depth beyond the SPA root.

Key Lessons

  • Context API scales well for this kind of orchestrated state - window management, theme switching, and music playback each got their own context without Redux overhead.
  • Framer Motion's AnimatePresence is indispensable for mount/unmount animations. Wrapping it around conditional renders is the cleanest pattern.
  • OS aesthetics need constraint. Every pixel matters when your UI is imitating a real operating system - inconsistency breaks the illusion.
  • Static HTML wins for SEO even inside an SPA project. Public-folder HTML pages are served directly and require zero JS to be indexed.
